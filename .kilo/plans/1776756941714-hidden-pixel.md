# Fix non-blocking CLI suggest picker: drop "Dismiss" row, fix misleading hints

## Context

PR #9298 made CLI suggestions render above the active main input. Two follow-up issues remain:

1. The picker still shows a redundant **"Dismiss"** row as a second option.
2. The footer hints advertise `↑↓ select  enter choose  esc dismiss`, but **none of those keys reach the picker** when the main input is focused:
   - `↑`/`↓` trigger prompt history navigation.
   - `enter` submits the main input.
   - `esc` does nothing in the picker.

## Root causes

### Why "Dismiss" appears

It is NOT LLM-generated. `packages/opencode/src/kilocode/suggestion/tui/prompt.tsx:12-15,27` hard-codes it:

```tsx
const dismiss = {
  label: "Dismiss",
  description: "Dismiss this suggestion and continue",
}
// …
const options = createMemo(() => [...props.request.actions, dismiss])
```

VS Code's equivalent (`packages/kilo-vscode/webview-ui/src/components/chat/SuggestBar.tsx:42-61`) does the opposite: it renders only `props.request.actions` as buttons and puts an **X (close) icon button** next to them for explicit dismissal. That's the model to follow.

### Why the keyboard hints lie

`packages/opencode/src/kilocode/suggestion/tui/prompt.tsx:64-66`:

```tsx
useKeyboard((evt) => {
  if (dialog.stack.length > 0) return
  if (props.nonBlocking && props.inputFocused?.()) return // ← swallows ALL keys
  // arrow / enter / escape / digit handlers below
})
```

The short-circuit kills the picker's entire keyboard handler whenever the main input is focused (i.e. always, in the new non-blocking mode). The `↑↓ select / enter choose / esc dismiss` footer is therefore misleading.

## What escape does everywhere else (so we know it's safe to repurpose)

Explore agent confirmed (see task `ses_250ce515effe0xu3JZ0eB80FBy`):

- Main prompt textarea: Escape is a **no-op** when autocomplete is closed and not in shell mode. It does NOT cancel generation, clear input, or exit the app. The raw event bubbles to every `useKeyboard` subscriber.
- Autocomplete open → textarea traits capture escape and `preventDefault`s it.
- Shell mode (`!` at col 0) → textarea handles escape and `preventDefault`s it.
- Dialog stack non-empty → dialog eats escape first; picker handler already early-returns when `dialog.stack.length > 0`.
- `question.tsx`, `permission.tsx`, `network.tsx` siblings in the footer are rendered mutually exclusively with `SuggestPrompt` (per `routes/session/index.tsx:1264-1297`), so no collision.
- `app_exit` (ctrl+c/ctrl+d/<leader>q) does NOT include escape (`config.ts:646`).

Conclusion: letting the picker's `useKeyboard` handler consume Escape in non-blocking mode is safe as long as we also guard against `evt.defaultPrevented` (for the autocomplete / shell-mode cases where the textarea already handled it).

## Plan

### 1. Remove the hard-coded "Dismiss" row

**File:** `packages/opencode/src/kilocode/suggestion/tui/prompt.tsx`

- Delete the `dismiss` constant (lines 12-15).
- Change line 27 from `[...props.request.actions, dismiss]` to `props.request.actions`.
- Simplify `choose` (lines 56-62): the out-of-bounds-index `reject()` path is now unreachable from the UI, so inline it to `choose(i) → accept(i)`. Keep the exported `reject` function — it's still called by the `escape` branch.
- Drop the `muted()` check (line 122) and its use (line 136) — it existed only to grey out the Dismiss row.

Net effect: picker renders exactly the LLM-provided actions (1 or 2 rows), matching VS Code's `SuggestBar`.

### 2. Let Escape through in non-blocking mode so it can dismiss

**Same file**, replace the short-circuit at line 66:

```tsx
if (props.nonBlocking && props.inputFocused?.()) return
```

with:

```tsx
if (evt.defaultPrevented) return
if (props.nonBlocking && props.inputFocused?.() && evt.name !== "escape") return
```

Effect:

- `↑`/`↓`/`j`/`k`/digits/`return` → still short-circuited, still go to main input (history, submit).
- `escape` → reaches the existing branch at line 98 (`if (evt.name === "escape" || keybind.match("app_exit", evt))`) and calls `reject()` → picker dismisses cleanly.
- `defaultPrevented` guard avoids double-handling when autocomplete or shell-mode already consumed the key.

Tighten the final branch slightly for correctness: the existing line 98 also matches `app_exit` (ctrl+c/ctrl+d/<leader>q). Those shouldn't dismiss a suggestion while the user is typing in the main prompt. Restrict the match to `evt.name === "escape"` only when in non-blocking-with-focused-input mode:

```tsx
if (evt.name === "escape" || (!(props.nonBlocking && props.inputFocused?.()) && keybind.match("app_exit", evt))) {
  evt.preventDefault()
  reject()
}
```

Simpler reformulation: split into two branches — one plain `escape` check (active in both modes), one `app_exit` check (active only in blocking mode).

### 3. Update the footer hints to match reality

**Same file**, the hints row at lines 158-168.

In non-blocking + input-focused mode, only `esc` works for keyboard; mouse click still chooses an action. So show: `click choose   esc dismiss`.

In blocking mode (unused by the `suggest` tool today but still a valid future caller path), keep the existing `↑↓ select   enter choose   esc dismiss`.

Implementation sketch:

```tsx
<box flexDirection="row" gap={2}>
  <Show
    when={props.nonBlocking && props.inputFocused?.()}
    fallback={
      <>
        <text fg={theme.text}>
          {"↑↓"} <span style={{ fg: theme.textMuted }}>select</span>
        </text>
        <text fg={theme.text}>
          enter <span style={{ fg: theme.textMuted }}>choose</span>
        </text>
      </>
    }
  >
    <text fg={theme.text}>
      click <span style={{ fg: theme.textMuted }}>choose</span>
    </text>
  </Show>
  <text fg={theme.text}>
    esc <span style={{ fg: theme.textMuted }}>dismiss</span>
  </text>
</box>
```

(`Show` is already imported via solid-js in this file — verify and add if needed.)

## Decisions (confirmed)

- **Footer hints in non-blocking mode:** `click choose   esc dismiss`.
- **Explicit dismiss path:** Escape dismisses the picker (section 2 above implements this by letting Escape through the short-circuit with a `defaultPrevented` guard).

## Out of scope

- Keyboard-only access to _accept_ a suggestion (e.g. Tab-to-focus-picker). Today keyboard-only users with a non-blocking picker have no way to accept without the mouse. That's a real limitation but a bigger UX change and not what these issues are asking about. Can be a follow-up.
- Changes to the `question` tool. It uses the same short-circuit pattern (`routes/session/question.tsx:136`) but questions are intentionally blocking and the user hasn't flagged them.
- Changes to blocking-mode behavior. The `suggest` tool always emits non-blocking now; blocking mode remains for future callers and its keyboard flow is untouched.
- Changes to `tool.txt` — the tool prompt never mentioned Dismiss; the LLM was never responsible for it.
